# Kiln on RK3588 (Radxa ROCK 5B) — LLM + vision NPU

**Status: SKELETON, NOT yet built or tested on real hardware.** The RK3588 path
is scaffolded the same way as the working RK3576 one (mainline kernel + Kiln
patches + the vendor `rknpu` module), and the Kiln device tree **compiles**, but
nothing has run on a ROCK 5B yet. Until it does, treat everything here as
"designed", not "works". The tested path is RK3576 (ROCK 4D) — see
[`MAINLINE-KERNEL.md`](MAINLINE-KERNEL.md).

> **Have a ROCK 5B and want to help?** The single most useful thing is to finish
> the bring-up: build the kernel with `kernel-patches-rk3588/` applied, boot the
> `rk3588-rock-5b-kiln.dtb`, and report whether the vendor NPU probes
> (`/dev/dri/renderD*`, `kiln-doctor`) — paste `kiln-doctor` + `dmesg` into a
> [GitHub issue](https://github.com/gahingwoo/kiln/issues). See the open items in
> [`../kernel-patches-rk3588/README.md`](../kernel-patches-rk3588/README.md).

## Why RK3588 is not a copy of RK3576

Both run the closed `librkllmrt` / `librknnrt` runtimes on the **vendor `rknpu`**
driver (out-of-tree, GPL), on a mainline kernel. The vendor `rknpu.ko` is the
**same binary** — it already carries the `rockchip,rk3588-rknpu` match and the
RK3588 config (`rk3588_rknpu_config`, `core_mask=0x7`, plus RK3588M/J leakage
binning) in the fetched `rknpu_drv.c`.

The difference is the **device tree**. On RK3576, mainline has no NPU node, so
Kiln adds one. On RK3588, **mainline already describes the NPU** — as three
`rockchip,rk3588-rknn-core` nodes for the open `accel/rocket` driver, and ROCK 5B
enables them. Kiln's runtimes need the vendor ioctl ABI, i.e. a **single**
`rockchip,rk3588-rknpu` node, which conflicts with the rocket cores over the same
MMIO. So the Kiln RK3588 DTB **deletes the three rocket cores and reshapes core 0
into the vendor node** (`rk3588-rock-5b-kiln.dts`).

## What's different from RK3576

| | RK3576 (ROCK 4D, tested) | RK3588 (ROCK 5B, skeleton) |
|---|---|---|
| NPU | dual "core" @ `27700000`, one vendor node | **triple core** @ `fdab0000/fdac0000/fdad0000`, one vendor node (`core_mask=0x7`) |
| `compatible` | `rockchip,rk3576-rknpu` | `rockchip,rk3588-rknpu` |
| Mainline NPU node | **none** — Kiln adds it | **already present** as rocket `rknn-core` (enabled on ROCK 5B) — Kiln deletes + reshapes |
| IOMMU | 2 banks | **3** (`fdab9000`+`fdaba000`, `fdaca000`, `fdada000`) |
| Power domains | NPUTOP → NPU0/NPU1 | **NPUTOP + NPU1 + NPU2** under `VD_NPU` |
| NPU clock | CRU `CLK_RKNN_DSU0` (Kiln left SCMI) | mainline uses **`SCMI_CLK_NPU`** — SCMI-vs-CRU needs board validation |
| CPU | A72+A53 (ARMv8.0) | **A76+A55 (ARMv8.2)** — `-mcpu=cortex-a76.cortex-a55` |
| Console | UART @ `0x2ad40000` | **UART2 @ `0xfeb50000`** (`serial2:1500000n8`) |
| Workload | LLM + vision (6 TOPS) | LLM + vision (6 TOPS, 3 cores) |
| u-boot | `rock4d-sd-uboot.img` (shipped) | `rock5b-sd-uboot.img` — **you supply it** |

## Open items (for the ROCK 5B bring-up)

The full, evidence-backed list is in
[`../kernel-patches-rk3588/README.md`](../kernel-patches-rk3588/README.md). In short:

1. **Reconcile the vendor node against a real RK3588 vendor BSP** — `reg`
   span/count, `clk_npu` route (SCMI vs CRU), OPP/voltage, leakage `nvmem-cells`.
2. **Which mainline kernel-patches apply** — the RK3576 `0001-0010` are
   RK3576-specific; RK3588 likely needs a different (smaller) set. Re-derive on
   hardware.
3. **On-board validation** — power-on, probe (`renderD*`), a correct conv on all
   3 cores, then LLM (`kiln-chat`) and the OpenAI server (`kiln-serve`).
4. **Vision model** — ship / point to a `mobilenetv2-12_rk3588.rknn`
   (`target_platform="rk3588"`; an rk3576 `.rknn` is rejected), or just convert
   on-board with `kiln-convert mobilenet` (it targets the running SoC).
5. **u-boot** — a ROCK 5B SD u-boot image for `post-image.sh` (Kiln ships none).

## Once it works

The userspace is identical to RK3576 — `kiln`, `kiln-chat`, `kiln-vision`,
`kiln-serve`, `kiln-convert`, `kiln-config`, `kiln-doctor` all work unchanged
(they go through `librkllmrt` / `librknnrt`, which handle RK3588). The install and
serve flows are the same as [`ARMBIAN.md`](ARMBIAN.md) and
[`SERVER.md`](SERVER.md).
